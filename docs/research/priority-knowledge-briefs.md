## C1_reversal_risk

**Summary:** Across 16 verified high-consequence Windows repair actions (of 19 originally scoped — 3 dropped for weak/single sourcing), registry, boot-configuration, and firewall/network tools have a documented built-in export/import undo and should only be classified "reversible" if that export step is actually run first. Disk-wipe-adjacent, TPM-clear, `DISM /ResetBase`, firmware-flash, and completed Reset-this-PC actions have **no software undo once run** — safety depends entirely on a prior backup, BitLocker-recovery-key escrow, or system image. System Restore and `Checkpoint-Computer` are the exception: they *are* the safety net, not actions that need one.

| Action | Risk Tier | Undo Path | Snapshot Before Running | Source |
|---|---|---|---|---|
| `chkdsk /r` | Reversible/corrective (safe to interrupt/re-run) | None single-command; re-run repairs remaining corruption | Copy accessible files off a failing drive first — FAT "discard lost chains" is non-undoable | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/chkdsk) |
| `DISM /StartComponentCleanup /ResetBase` | Destructive (ResetBase) | None — "All existing update packages can't be uninstalled after this command" | System Restore point / full image before `/ResetBase` | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-hardware/manufacture/desktop/clean-up-the-winsxs-folder?view=windows-11) |
| `pnputil /delete-driver` | Destructive by default, reversible if prepared | Re-`add-driver` from an exported copy | `pnputil /export-driver` before deleting | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-hardware/drivers/devtest/pnputil-command-syntax) |
| TPM Clear (`tpm.msc`) | Destructive | None — cleared keys unrecoverable | Escrow/confirm BitLocker recovery key, suspend BitLocker, clear via OS tool not UEFI | [Microsoft Learn](https://learn.microsoft.com/en-us/windows/security/hardware-security/tpm/initialize-and-configure-ownership-of-the-tpm) + [BitLocker FAQ](https://learn.microsoft.com/en-us/windows/security/operating-system-security/data-protection/bitlocker/faq) |
| `bootrec /fixmbr /fixboot /rebuildbcd` | Reversible if backup step followed | `bcdedit /import` (rebuildbcd only); fixmbr/fixboot simply overwrite, no undo | `bcdedit /export`, rename (not delete) existing BCD store first | [Microsoft Learn](https://learn.microsoft.com/en-us/troubleshoot/windows-client/performance/windows-boot-issues-troubleshooting) |
| `bcdedit` | Reversible if exported first | `bcdedit /import <file>` | `bcdedit /export <file>` before changes | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-hardware/manufacture/desktop/bcdedit-command-line-options?view=windows-11) |
| `bcdboot` | Destructive to existing boot-entry customizations by default | `/m` merge flag prevents erasure; otherwise none | `bcdedit /enum all /v` to record entries; confirm correct system-partition letter | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-hardware/manufacture/desktop/bcdboot-command-line-options-techref-di?view=windows-11) |
| `netsh int ip reset` | Reversible (config only, no data loss) | None dedicated; manually re-enter static IP/DNS | Record current config (`ipconfig /all`); command auto-logs to resetlog.txt | [Microsoft Learn](https://learn.microsoft.com/en-us/troubleshoot/windows-server/networking/reset-tcp-ip-net-shell) |
| `netsh winsock reset` | Reversible (3rd-party LSPs need reinstall) | Re-run affected software installers | `netsh winsock dump > backup.txt` first | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/netsh-winsock) |
| `netsh advfirewall reset` | Reversible if exported first | `netsh advfirewall import <file>` | `netsh advfirewall export <file>` (or combined `reset export`) | [Microsoft Learn](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/netsh-advfirewall) |
| DDU (Display Driver Uninstaller) | Reversible only if vendor precaution followed | None built-in | Vendor-recommended System Restore point/backup + run in Safe Mode | [Wagnardsoft (vendor)](https://www.wagnardsoft.com/content/Download-Display-Driver-Uninstaller-DDU-18154) |
| Reset this PC | Destructive | None once complete | Back up files; capture BitLocker recovery key first | [Microsoft Support](https://support.microsoft.com/en-us/windows/reset-your-pc-0ef73740-b927-549b-b7c9-e6f2b48d275e) |
| Firmware/BIOS flash | Destructive | None generally; vendor recovery jumper/USB if board survives | Full backup, stable AC/UPS power, suspend BitLocker, verify exact model match | [Dell KB](https://www.dell.com/support/kbdoc/en-us/000124211/dell-bios-updates) + [BitLocker FAQ](https://learn.microsoft.com/en-us/windows/security/operating-system-security/data-protection/bitlocker/faq) |
| System Restore (`rstrui`) | Reversible (safety net itself) | Built-in "Undo System Restore" (booted session only) | N/A — is the snapshot mechanism; doesn't cover personal files | [Microsoft Support](https://support.microsoft.com/en-us/windows/experience/backup-recovery/system-restore) + [MS Q&A](https://learn.microsoft.com/en-us/answers/questions/cd7d2546-b975-4254-8e45-d8e3771c3f5c/undo-system-restore?forum=windows-all) |
| `Checkpoint-Computer` | Read-only/safe | N/A — nothing destructive to reverse | This command IS the snapshot step (run before risky actions; 24hr cooldown) | [Microsoft Learn](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.management/checkpoint-computer?view=powershell-5.1) |
| `Optimize-Volume` | Mixed: `-Defrag` low risk / `-ReTrim` destructive (SSD default) | None for `-ReTrim` — TRIMmed sectors purged | Recover any needed deleted files from free space *before* running (or before scheduled Optimize Drives task fires) | [Microsoft Learn](https://learn.microsoft.com/en-us/powershell/module/storage/optimize-volume?view=windowsserver2025-ps) |

**Open/uncertain:** 3 of the original 19 actions were dropped for insufficient sourcing (not identified beyond the count) — treat the matrix as strong but not fully complete. DDU is non-Microsoft freeware (Wagnardsoft), included only because the task named it. Firmware-flash guidance uses Dell as one representative OEM; HP/Lenovo/motherboard-vendor steps differ in detail though the AC-power/no-interrupt/backup pattern holds. Two entries (System Restore's undo path, `diskpart clean-all`'s irrecoverability) lean on a Microsoft Q&A community post as secondary corroboration rather than a primary reference article. Several "reversible" verdicts (bootrec, bcdedit, netsh resets, DDU) are conditional — skipping the documented export/backup step converts them to destructive in practice.

## D1_liability_consent

> **Not legal advice.** General research only — have a licensed attorney review any consent/waiver language and liability allocation before deploying it in a customer-facing contract. State law varies; the sources below deliberately mix federal statutes with California/Texas examples to show the framework's shape, not a jurisdiction-specific answer.

**Summary:** For a shop running autonomous, potentially destructive repairs, liability and consent rests on: (1) computer-crime law making the customer's *scoped* authorization the line between lawful repair and a crime; (2) UCC §2-719(3) permitting enforceable consequential-damages caps/disclaimers for data loss (unless unconscionable), which real shop contracts (Geek Squad, etc.) implement by shifting backup duty to the customer; and (3) bailment/warehouseman law (UCC §7-204) establishing an "ordinary care" duty for a device in the shop's possession — limitable by contract, but not eliminable.

| Finding | Detail | Sources (2 independent) |
|---|---|---|
| Consent must be scoped, not assumed | CFAA (18 U.S.C. §1030) criminalizes access "without authorization" or "exceeding authorized access"; Texas's Breach of Computer Security statute requires the owner's "effective consent." A vague "fix my computer" intake does not obviously authorize every destructive action an autonomous system might take. | [Cornell LII — 18 U.S.C. §1030](https://www.law.cornell.edu/uscode/text/18/1030) + [FindLaw — Tex. Penal Code §33.02](https://codes.findlaw.com/tx/penal-code/penal-sect-33-02/) |
| Data-loss liability can be capped/disclaimed | UCC §2-719(3): consequential-damages exclusions/caps enforceable unless unconscionable; commercial-loss caps (vs. consumer personal-injury caps) are not presumptively unconscionable. Geek Squad's real contract implements this — customer bears backup duty, shop's remedy capped at a small fixed sum absent a paid backup add-on. | [Cornell LII — UCC §2-719](https://www.law.cornell.edu/ucc/2/2-719) + [Best Buy/Geek Squad service contract](https://storage.bestbuy.com/geeksquad/terms/geek_squad_instore_servicecenter_repair/geek_squad_instore_servicecenter_repair_2023_02_01_to_current_english.pdf) |
| Duty of care while device is in shop possession | Accepting a device for repair forms a bailment; UCC §7-204 sets an "ordinary/reasonable care" standard for a bailee-for-hire, limitable by contract term (not against public policy) but not eliminable — cannot excuse conversion (losing/misappropriating the device). | [Cornell LII — UCC §7-204](https://www.law.cornell.edu/ucc/7/7-204) + [UpCounsel — Bailment of Goods](https://www.upcounsel.com/bailment-of-goods) |

**Open/uncertain (3 dropped, plus explicit research gaps):**
- No PC/electronics-repair-specific statute was found that squarely addresses **autonomous/AI-driven** destructive actions. California's B&P Code §9844 (Electronic and Appliance Repair Dealer law) is the closest analog but was written for human technicians and general repairs — treat any statute here as a floor; give customers itemized, action-specific consent (e.g., a separate signature for "wipe drive") beyond what any statute strictly requires.
- CFAA/state computer-crime law is not squarely litigated against repair shops acting on the customer's own request — it's cited as the doctrinal reason consent must be scoped, not as a body of repair-shop case law.
- Geek Squad/industry contract language is commercial custom, not binding precedent; enforceability of any specific liability clause still turns on state law (many states carve out gross negligence/willful misconduct from waivers — a secondary source on this point returned a fetch error and was excluded rather than cited unverified).
- State right-to-repair laws (NY Digital Fair Repair Act, Colorado HB24-1121) were reviewed but excluded — they govern manufacturer-to-repairer parts/documentation duties, not shop-to-customer consent/liability, and are adjacent rather than on-point.

## D2_privacy_law

> **Not legal advice.** General research only — applicability to a specific shop's card-data handling, health-adjacent data, or multi-state customer base is fact-specific; confirm with counsel and, for payment-card questions, a QSA.

**Summary:** A PC-repair shop is not categorically exempt from any privacy regime: baseline data-minimization/disposal norms come from FTC guidance (echoed as a binding CCPA/CPRA standard for covered California businesses); breach notification is state-law-driven with real timing differences (CA 30 days/500-resident AG threshold vs. TX 60 days/250); PCI DSS is triggered by function (any storage/processing/transmission of card data), not size or intent; and HIPAA applies only to covered entities/business associates — a shop that merely repairs a device with incidental PHI exposure does not become regulated, but one contracted by a covered entity to maintain/back up PHI-bearing systems does.

- **Data minimization & disposal:** FTC business guidance discourages retaining sensitive data without need; CCPA/CPRA regulations require collection/retention be "reasonably necessary and proportionate." NIST SP 800-88 + FTC disposal guidance require rendering disposed media unreadable, not just deleted. — [FTC](https://www.ftc.gov/business-guidance/resources/protecting-personal-information-guide-business) + [Cornell LII — 11 CCR §7002](https://www.law.cornell.edu/regulations/california/11-CCR-7002); [FTC disposal](https://www.ftc.gov/business-guidance/resources/disposing-consumer-report-information-rule-tells-how) + [NIST SP 800-88r2](https://csrc.nist.gov/pubs/sp/800/88/r2/final)
- **CCPA/CPRA applicability:** Covers a business only above $25M revenue, 100,000+ consumers/households of data annually, or 50%+ revenue from data sales — a small independent shop is unlikely to qualify; a regional/franchise operation could. — [Cal. Civ. Code §1798.140](https://leginfo.legislature.ca.gov/faces/codes_displaySection.xhtml?lawCode=CIV&sectionNum=1798.140) + [IAPP](https://iapp.org/news/a/does-the-ccpa-as-modified-by-the-cpra-apply-to-your-business)
- **CCPA private right of action for breaches:** $100–$750/incident (or actual damages) for breach caused by failure to maintain reasonable security, subject to a 30-day cure notice. — [Cal. Civ. Code §1798.150](https://leginfo.legislature.ca.gov/faces/codes_displaySection.xhtml?lawCode=CIV&sectionNum=1798.150.) + [ConsumerPrivacyAct.com](https://www.consumerprivacyact.com/section-1798-150-private-right-of-action/)
- **Breach notification varies by state:** CA requires notice within 30 days, AG notice if 500+ residents affected; TX allows 60 days, AG notice at 250+ residents; all 50 states + DC now have their own statute. — [Cal. Civ. Code §1798.82](https://leginfo.legislature.ca.gov/faces/codes_displaySection.xhtml?lawCode=CIV&sectionNum=1798.82) + [CA AG](https://oag.ca.gov/privacy/databreach/reporting); [Tex. Bus. & Com. Code §521.053](https://codes.findlaw.com/tx/business-and-commerce-code/bus-com-sect-521-053/) + [Texas.public.law](https://texas.public.law/statutes/tex._bus._and_com._code_section_521.053); [Privacy Rights Clearinghouse 50-state survey](https://privacyrights.org/resources-tools/reports/data-breach-notification-laws-50-state-survey-2026-edition)
- **Other comprehensive state laws (volume-based, no revenue floor):** Virginia CDPA and Colorado CPA both apply at 100,000+ consumers/year (or 25,000+ with data-sale revenue) regardless of company revenue. — [Va. Code Title 59.1 Ch. 53](https://law.lis.virginia.gov/vacodefull/title59.1/chapter53/) + [Privacy Rights Clearinghouse](https://privacyrights.org/resources-tools/law-overviews/virginia-consumer-data-protection-act); [Colo. Rev. Stat. §6-1-1304](https://colorado.public.law/statutes/crs_6-1-1304) + [FindLaw](https://codes.findlaw.com/co/title-6-consumer-and-commercial-affairs/co-rev-st-sect-6-1-1304/)
- **PCI DSS triggers on function, not intent:** Any entity storing/processing/transmitting cardholder data (PAN) or sensitive authentication data (CVV, PIN, track data — never storable post-authorization) is in scope. A shop taking card payments is in scope for its own POS; a shop that recovers a PAN from a customer's failed drive pulls that drive into PCI scope. — [PCI SSC Glossary](https://www.pcisecuritystandards.org/glossary/) + [pcidssguide.com](https://pcidssguide.com/pci-dss-requirement-3/); [PCI SSC FAQ on retention](https://www.pcisecuritystandards.org/faq/articles/Frequently_Asked_Question/what-is-the-maximum-period-of-time-that-cardholder-data-can-be-stored/)
- **HIPAA applies only to covered entities/business associates:** A generic walk-in repair shop is neither by default, and incidental/transient PHI exposure during repair does not create business-associate status (HHS FAQ 243, "conduit"-style exception) — but a shop contracted by a covered entity to maintain/back up/recover PHI-bearing data takes on business-associate status and needs a signed BAA. — [HHS.gov](https://www.hhs.gov/hipaa/for-professionals/covered-entities/index.html) + [HIPAA Journal](https://www.hipaajournal.com/covered-entities-under-hipaa/); [HHS FAQ 243](https://www.hhs.gov/hipaa/for-professionals/faq/243/is-a-business-associate-contract-required-for-inadvertent-contact-with-phi/index.html) + [Hawley Troxell](https://hawleytroxell.com/insights/do-you-have-valid-business-associate-agreements/)
- **GLBA Safeguards Rule generally doesn't apply:** Targets "financial institutions" by activity (tax prep, mortgage brokerage, in-house financing) — a pure hardware/software repair shop is normally out of scope unless it also extends credit or performs a listed financial activity. — [FTC](https://www.ftc.gov/business-guidance/resources/ftc-safeguards-rule-what-your-business-needs-know) + [Cornell LII 16 CFR §314.6](https://www.law.cornell.edu/cfr/text/16/314.6)
- **FTC Section 5 as a backstop:** Even absent a sector-specific law, FTC "unfairness" authority and state UDAP statutes impose a general "reasonable security" expectation regardless of business size. — [FTC small-business guide](https://www.ftc.gov/business-guidance/resources/start-security-guide-business) + [SBA](https://www.sba.gov/business-guide/manage-your-business/stay-safe-cybersecurity-threats)

Several official `.gov` pages (eCFR direct, NCSL, coag.gov, HHS FAQ 244, Colorado AG, and a handful of PDFs) blocked automated fetching and were sourced instead via mirror/reproduction sites (Cornell LII, FindLaw, Justia-style, public.law) — treated as authoritative reproductions, not substitutes for reading the primary text. ## E1_local_llm

**Summary:** For a 24–48GB VRAM home workstation with no cloud dependency, dense 24–32B models (Qwen2.5/Qwen3-32B, Mistral-Small-3.2-24B, Gemma-3-27B) at 4-bit GGUF/AWQ fit a single 24GB card as a "judge," paired with a cheaper 8–14B diagnosis model; MoE models (Qwen3-30B-A3B, Llama-4-Scout, DeepSeek-R1-Distill-Qwen-32B) trade VRAM for speed via sparse activation. A 48GB tier (2×24GB or one RTX 6000 Ada) unlocks 70–72B dense models at 4-bit and Llama-4-Scout at int4; two RTX 3090s running Qwen3.6-27B at AWQ-INT4 via vLLM reportedly reach ~100 tok/s.

- **Serving stacks:** `llama.cpp` (GGUF, k-quant down to 1.5-bit, CPU+GPU hybrid offload) — [GitHub](https://github.com/ggml-org/llama.cpp). `vLLM` (PagedAttention, continuous batching, AWQ/GPTQ/GGUF/FP8, tensor-parallel multi-GPU) — [vLLM docs](https://docs.vllm.ai/en/latest/). `LM Studio` (GUI wrapper over llama.cpp/MLX) — [LM Studio docs](https://lmstudio.ai/docs/app). `Ollama` (llama.cpp wrapper with VRAM-aware auto-offload scheduling) — [Ollama docs](https://docs.ollama.com/gpu).
- **24GB-tier judge candidates:** Qwen2.5-32B-Instruct GGUF — Q4_K_M 19.9GB (fits with headroom), Q5_K_M 23.3GB (tight), Q8_0 34.8GB (needs 48GB tier) — [HF model card](https://huggingface.co/Qwen/Qwen2.5-32B-Instruct-GGUF). Qwen3-32B dense / Qwen3-30B-A3B MoE (3B active) — [QwenLM GitHub](https://github.com/QwenLM/Qwen3). Gemma-3-27B, 128K context, QAT checkpoints for consumer GPUs — [Google AI model card](https://ai.google.dev/gemma/docs/core/model_card_3). Mistral-Small-3.2-24B — strong native function-calling, needs ~55GB at bf16 so requires 4-bit quant on a 24GB card — [HF model card](https://huggingface.co/mistralai/Mistral-Small-3.2-24B-Instruct-2506). DeepSeek-R1-Distill-Qwen-32B — chain-of-thought reasoning distilled from full R1, good "judge" fit — [HF model card](https://huggingface.co/deepseek-ai/DeepSeek-R1).
- **48GB-tier / MoE options:** Llama-4-Scout (109B total/17B active) — Meta states fits a single H100 at int4; realistically a 48GB-tier target with aggressive quant, not a 24GB fit. Maverick (400B/17B active) is out of reach for any home workstation. — [Meta AI blog](https://ai.meta.com/blog/llama-4-multimodal-intelligence/). Newer Qwen3.5/Qwen3.6 (Feb/Apr 2026) MoE-hybrid line, 262K–1.01M context — [QwenLM GitHub](https://github.com/QwenLM/Qwen3.6).
- **Quantization:** AWQ is the leading GPU-served INT3/INT4 alternative to GGUF, ~2.7x speedup over FP16 on RTX 4090 for an 8B model — [AWQ GitHub](https://github.com/mit-han-lab/llm-awq). Qwen's own benchmark shows AWQ-INT4 roughly doubling-to-tripling decode throughput vs BF16 on the same hardware — [Qwen speed benchmark docs](https://qwen.readthedocs.io/en/latest/getting_started/speed_benchmark.html).
- **Concrete throughput data points (community-benchmarked):** RTX 4090, Q4_K_XL — 8B ~131 tok/s, 14B ~83 tok/s, 30B MoE ~196 tok/s ("sweet spot"), dense 32B only ~39 tok/s — [Hardware Corner](https://www.hardware-corner.net/rtx-4090-llm-benchmarks/). Dual RTX 3090 (48GB total), Qwen3.6-27B AWQ-INT4 via vLLM, tensor-parallel-size 2, FP8 KV cache, full 256K context — ~100 tok/s — [Chris Dzombak blog](https://www.dzombak.com/blog/2026/04/a-vllm-docker-compose-recipe-for-running-qwen-3-6-27b-on-dual-rtx-3090s-opencode-configuration/).
- **Reference GPU specs:** RTX 4090 = 24GB GDDR6X; RTX 6000 Ada = 48GB GDDR6 ECC — [NVIDIA RTX 4090](https://www.nvidia.com/en-us/geforce/graphics-cards/40-series/rtx-4090/) + [NVIDIA RTX 6000 Ada](https://www.nvidia.com/en-us/products/workstations/rtx-6000/).
