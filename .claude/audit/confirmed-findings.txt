[C1] HIGH  —  attestation_message uses unescaped separators, enabling a same-signature collision between structurally different contributions (signed-byte ambiguity)
====================================================================================================
LOCATION: crates/corpus-client/src/schema.rs:230-262 (attestation_message); the verify side is crates/corpus-client/src/gate.rs:114 (ensure_attested -> authority.verify(&attestation_message(contribution), ...)). Contrast the injection-safe pattern at crates/provenance/src/lib.rs:104-111 (canonical length-prefixes description).

REASONING:
The finding is real and I reproduced a concrete collision. attestation_message (schema.rs:230-262) hand-builds the only bytes the ed25519 sign-off signature covers, using ':' ',' ';' '=' as structural separators and '\n' (via writeln!) as the record terminator, and interpolates every field UNESCAPED and UN-length-prefixed. All the interpolated fields are arbitrary, unvalidated Strings: o.signature.symptoms (Symptom(pub String), fault.rs:5), o.plan.id, step.action (line 240), config_class.key() (BomRevision(String), config_class.rs), p.run_id and primed_from (lines 252-258), and part_class via label_tag (lines 291-292). I grepped the entire submit/attest/verify path (schema.rs, store.rs, gate.rs) — there is NO validation that rejects separator/newline bytes in any of these fields before signing or verifying. This directly contradicts the doc comment's 'unambiguous' claim and contrasts with provenance::canonical (lib.rs:96-114), which deliberately length-prefixes free text for exactly this reason.

Because the encoding is ambiguous, two structurally different contributions can serialize to identical signed bytes, so a single ed25519 signature legitimately produced for benign contribution A is a VALID attestation for malicious contribution B (gate.rs:114 recomputes attestation_message(B) from B's typed fields and verifies sig_A against it — verification succeeds, B is admitted as authority-signed truth).

I built and ran an exact mirror of attestation_message and proved a byte-for-byte collision (output: 'COLLIDE (bytes equal): true'):
  A (what an honest authority reviews and signs — a READ-ONLY, human-approved fix): plan.id = "p1\nstep:wipe_disk:Destructive", steps = [inspect/ReadOnly]. Plan::risk() = ReadOnly.
  B (what is actually admitted): plan.id = "p1", steps = [wipe_disk/Destructive, inspect/ReadOnly]. Plan::risk() = Destructive.
Both render to:
  ...\nplan:p1\nstep:wipe_disk:Destructive\nstep:inspect:ReadOnly\nlabel:resolved_confirmed\nsignoff:HumanConfirmed\nclass:bom-1\nprov:none\n
— identical bytes. So a human who approved a benign read-only plan unknowingly produces a signature that admits a DESTRUCTIVE plan into the corpus as a human-confirmed fix. ensure_evidence_integrity's destructive-needs-human check (gate.rs:81-85) does not help: both rows carry HumanConfirmed, and the destructive plan B passes the gate. This defeats the core 'asserting party != approving party' integrity property the whole ed25519 layer exists to provide (lib.rs:136-148).

Note the injection is ADDITIVE (writeln! always appends B's real typed lines too), so the finding's toy example (plan.id = "p1\nstep:rm_rf:Destructive\nlabel:resolved_confirmed" colliding with a row that 'genuinely has that step/label') is imprecise about the exact byte layout — the injected lines must equal B's genuinely-emitted prefix lines, as in my A/B construction. The mechanism it describes (unescaped separators/newlines collapsing distinct structures to the same signed bytes) is nonetheless correct, and I produced a working instance. The run_id/primed_from anti-replay binding (lines 252-258) is similarly defeatable by the same class of crafting (e.g. migrating step/label/class structure across the boundary while holding run-provenance fields free to vary), so the documented 'one valid attestation cannot be replayed onto clones with fabricated run ids' is not actually guaranteed by the bytes.

The chain_hash at schema.rs:269-283 uses serde_json (JSON-escaped, injection-safe) and JSONL persistence escapes \n to \\n, so a poisoned plan.id round-trips cleanly through the file — there is no incidental mitigation. Severity high is correct: it is a cryptographic-integrity bypass of the truth-admission gate, but it requires the attacker to obtain a real authority signature over a chosen (benign) A first, so it is not an unauthenticated forgery — high, not critical.

FIX:
Make attestation_message injection-proof by length-prefixing (or hashing) every interpolated value, mirroring the existing provenance::canonical pattern (lib.rs:104-111). For each variable field emit its byte length before its content, e.g.:
  writeln!(s, "plan[{}]:{}", o.plan.id.len(), o.plan.id);
  for step in &o.plan.steps { writeln!(s, "step:act[{}]={};risk={:?}", step.action.len(), step.action, step.risk); }
  writeln!(s, "class[{}]:{}", c.config_class.key().len(), c.config_class.key());
and length-prefix each symptom element (length-prefix per element, not a comma-join) and run_id / each primed_from id and part_class. Length-prefixing the variable-count and free-string fields removes the ambiguity that lets bytes migrate across record boundaries. Bump the domain tag to v3 since the encoding changes. As defense-in-depth, also reject any of these fields containing '\n' before signing/verifying. Add a regression test asserting attestation_message differs for the A/B pair above (a one-ReadOnly-step plan with a newline-poisoned id vs a two-step destructive plan with a clean id).

[C2] MEDIUM  —  config_class type confusion: BomRevision(x) and DerivedHash(x) attest identically, letting a valid attestation be replayed across comparability classes
====================================================================================================
LOCATION: crates/corpus-client/src/schema.rs:244 (the `class:` line in `attestation_message`), rooted in crates/common/src/config_class.rs:45-50 (`ConfigClass::key` returns the inner string for BOTH variants). Comparison divergence at crates/corpus-client/src/store.rs:84 (`row.config_class != *config_class`).

REASONING:
Traced the full path in the current source. `attestation_message` (schema.rs:230-262) builds the signed bytes manually and binds config_class only via `c.config_class.key()` at line 244. `ConfigClass::key()` (config_class.rs:45-50) returns the inner `String` for both `BomRevision(s)` and `DerivedHash(s)`, so `BomRevision("abc")` and `DerivedHash("abc")` produce byte-identical attestation messages. `ensure_attested` (gate.rs:104-119) re-derives those exact bytes and calls `authority.verify(...)`, so a signature minted for one variant verifies unchanged for the other. Meanwhile `fix_mappings` (store.rs:82-87) compares with `row.config_class != *config_class`, where `ConfigClass` derives `PartialEq` over the whole variant (config_class.rs:10), so the two are treated as DISTINCT comparability classes for retrieval.

Exploit path holds: take a legitimately-attested `DerivedHash("abc")` contribution, flip the variant to `BomRevision("abc")` (both variants are public; the same key string is reused), keep the `signature`. `admit` (store.rs:14-23) runs only `ensure_evidence_integrity` + `ensure_attested` — the attestation still verifies because the message is unchanged — so the row is admitted and served under a comparability class the authority never approved it for. I checked whether the hash chain rescues this: it does NOT. `chain_hash` (schema.rs:269-283) does serialize the full row including config_class, but the chain is recomputed fresh on every `submit` (store.rs:333-352) and re-verified relative to itself on `open` (verify_chain, store.rs:136-158); an attacker who writes the row simply re-chains it, and the chain still verifies. The attestation is the ONLY authority-bound integrity over config_class, and it is exactly the broken surface. So the attested context (which comparability class the authority signed for) is not actually bound — a real integrity bypass of the attestation boundary.

Severity medium is correct. It is a genuine attestation-replay/integrity-boundary defect, but practical exploitability is constrained: the attacker needs control of contribution submission AND a key string that is meaningful in both variants (most realistically constructing `BomRevision(<64-hex>)` matching a real derived hash, or `DerivedHash(<bom-string>)`), and the mis-routing is between two classes that share the same key string. Not refutable — the code does not handle it; not low — it defeats a deliberately-built cryptographic admission boundary.

FIX:
Bind the variant discriminant in the canonical message. In `attestation_message` (schema.rs:244) replace `writeln!(s, "class:{}", c.config_class.key())` with a tag that encodes the variant, e.g. add a helper on `ConfigClass`:

    pub fn tagged_key(&self) -> String {
        match self {
            ConfigClass::BomRevision(r) => format!("bom:{r}"),
            ConfigClass::DerivedHash(h) => format!("derived:{h}"),
        }
    }

and emit `writeln!(s, "class:{}", c.config_class.tagged_key())`. Because the message constant is already `cec-signoff-attestation-v2`, either bump it to `-v3` or treat this as part of the v2 rollout; old signatures over the bare key will no longer verify (intended — they were ambiguous). The `HttpCorpus::query` URL at store.rs:393 and the BomRevision/DerivedHash collision there is a separate routing concern and is out of scope for this attestation fix.

[C3] MEDIUM  —  Attestation does not bind outcome.verification, the very evidence the gate keys 'resolved' on
====================================================================================================
LOCATION: crates/corpus-client/src/schema.rs:230-262 (attestation_message — omits outcome.verification) vs crates/corpus-client/src/gate.rs:65-78 (ensure_evidence_integrity, resolved->matching-pass check on outcome.verification.result)

REASONING:
Verified against the real source (line numbers in the finding were off — gate.rs is 266 lines, the resolved/pass check is at gate.rs:65-78, not 514-527 — but the substance holds exactly).

attestation_message (schema.rs:230-262) builds the signed bytes from: signature.fingerprint, sorted symptoms, plan.id + each step (action+risk), label_tag, sign_off, config_class.key(), and provenance (run_id/retrieval_first/sorted primed_from). It does NOT include outcome.verification (result, class, or recurring). provenance::SignOffAuthority::attest / SignOffPublicKey::verify (lib.rs:189, 222) operate purely on those bytes; there is no separate binding of the verdict anywhere.

The gate, however, keys the entire 'resolved' admission on outcome.verification.result: ensure_evidence_integrity (gate.rs:65-78) requires ResolvedConfirmed => Some(Pass) and ResolvedProvisional => Some(ProvisionalPass). So the cryptographic anchor proves the authority saw the label but says nothing about the verdict that justifies it.

Concrete failure path: An authority signs a row R with label=ResolvedConfirmed while outcome.verification is None (or Fail). The signature is valid over the v2 message. A submitter/embedder who holds that attestation sets outcome.verification = Some(Verification::pass()) and leaves attestation untouched. At submit, admit() runs ensure_evidence_integrity (label ResolvedConfirmed + Pass match -> OK, gate.rs:71) and ensure_attested (recomputes attestation_message, which is unchanged because verification is excluded -> verify() returns true, gate.rs:114). Both gates pass; the authority never attested the Pass.

Adversarial check of mitigations — none save it: (1) FileCorpus's hash chain (chain_hash, schema.rs:269-283) does cover the full serialized contribution including outcome.verification, but it is computed by the store at WRITE time, AFTER admission (store.rs:333-340), over whatever verdict the submitter already supplied. It detects post-write hand edits, not a pre-submit fabricated verdict, so it does not protect this field at admission. (2) LocalCorpus and HttpCorpus have no chain at all. (3) The existing replay/tamper tests (store.rs:736-772) only cover sign_off and provenance, which ARE in the message — none cover verification, so the gap is untested.

The doc comments overstate the binding: schema.rs:225 claims 'the gate re-derives exactly what the authority signed' and schema.rs:89-93 claims the verdict is 'bound to the row so a resolved label can be ... gated on the evidence', yet the signed tuple omits that very verdict. The label and the verdict are cryptographically decoupled.

Severity: agree with the reported medium. This is the system's strongest integrity claim and it fails to cover the most security-critical evidence (the passing verdict), so it is a genuine integrity gap, not a style nit. It is not higher than medium because exploitation requires the attacker to already hold a valid authority attestation over a row the authority labeled Resolved* — an authority signing ResolvedConfirmed is already vouching for resolution — so the practical decoupling is between 'resolved label' and 'specific passing Verification record' rather than a full unsigned-row bypass.

FIX:
Bind the verification verdict into the signed bytes in attestation_message (schema.rs), and bump the domain tag so old signatures don't validate under the new schema. After the label/class/prov writes, add a stable encoding of outcome.verification, e.g.:

    let _ = writeln!(s, "label:{}", label_tag(&o.label));
    match &o.verification {
        None => { let _ = writeln!(s, "ver:none"); }
        Some(v) => {
            let mut rec: Vec<&str> = v.recurring.iter().map(|x| x.0.as_str()).collect();
            rec.sort_unstable();
            let _ = writeln!(s, "ver:{:?};class={:?};rec={}", v.result, v.class, rec.join(","));
        }
    }

and change the header from "cec-signoff-attestation-v2\n" to "...-v3\n". (Verification's fields are result: VerificationResult, class: Option<VerificationClass>, recurring: Vec<Symptom> — common/src/verification.rs:64-76 — so all three are available.) The legit flow already calls .attested_by() after constructing the Outcome with its verification set, so no caller change is needed. Add a regression test mirroring tampering_with_the_tuple_after_attestation_is_refused: attest a row, then mutate outcome.verification and assert GateError::AttestationInvalid.

[C4] HIGH  —  FileCorpus::open re-admits rows on only an unkeyed hash chain — attestation is never re-checked at open, so a file editor can serve forged "confirmed fixes"
====================================================================================================
LOCATION: crates/corpus-client/src/store.rs:265-290 (FileCorpus::open) and :136-158 (verify_chain); the gate/admit boundary is invoked only at :239, :330, :417 (the submit paths). chain_hash is unkeyed sha256 over public inputs at crates/corpus-client/src/schema.rs:269-283. Real wiring: crates/support-agent/src/main.rs:322-337 (open → with_authority → query).

REASONING:
Traced the concrete failure path in the current source and could not refute it.

1) Admission boundary is write-only. admit() (store.rs:14-23) calls ensure_evidence_integrity + ensure_attested. It is invoked ONLY from the three submit() impls (store.rs:239 LocalCorpus, :330 FileCorpus, :417 HttpCorpus). FileCorpus::open (store.rs:265-290) runs solely verify_chain(&rows) at :282 and never calls admit()/ensure_attested(). query()/fix_mappings (store.rs:318-325, :64-129) perform zero attestation verification — they filter by signature/config_class and aggregate confirmations. So rows arriving via the file are served with no attestation check.

2) The chain is tamper-evident, not tamper-resistant. chain_hash (schema.rs:269-283) is plain sha256 over (prev || "\n" || serde_json(row-with-integrity-cleared)). Every input is public; there is no secret and no authority signature over the head. verify_chain (store.rs:136-158) only recomputes that same public function. An attacker with write access to the .jsonl edits/inserts any row (e.g. flips an EscalatedHumanUnresolved row into ResolvedConfirmed carrying Verification::pass(), or injects a fabricated row), then recomputes prev/hash forward over every later row. open() then accepts the file and query() serves the forged plan.

3) The defeat holds even with an authority configured. open() hardcodes authority: None (store.rs:286); main.rs:322-325 calls FileCorpus::open(path) FIRST, then .with_authority(pubkey) AFTER rows are already loaded and accepted, then queries at :337. So --signoff-pubkey gates only future submits; at-rest rows that drive retrieval (and ranking by mapping.confirmations at main.rs:352-360) are never attestation-checked. This directly defeats the documented invariant "the engine cannot mint a confirmed row" for the at-rest store, which RowIntegrity's doc (schema.rs:98-104) and FOLLOWUPS acknowledge only tail-truncation as the residual — the full re-chaining and open-time attestation gap are undocumented.

The two test "refutations" do not cover this: file_corpus_refuses_a_hand_edited_row (store.rs:787-808) only edits "boot_loop"→"tampered_symptom" WITHOUT recomputing the chain, so it exercises the naive-editor case, not a re-chained edit; and the attestation tests (697-772) all go through submit(), never through open().

Precondition: write access to the local corpus file. That is a real precondition, but it is exactly the threat the ed25519 attestation design exists to neutralize, and the at-rest store fully bypasses it. High is appropriate; not critical because it requires local file write rather than being remotely exploitable.

FIX:
On open(), after verify_chain succeeds, run the admission gate over every loaded row before accepting the file — at minimum ensure_attested when an authority will be configured. Because open() currently runs before with_authority(), restructure so the authority is known at load time: e.g. add FileCorpus::open_with_authority(path, Option<SignOffPublicKey>) (or have with_authority re-validate the already-loaded rows and fail the build), and in that path call `for row in &rows { admit(row, &authority)?; }`. attestation_message (schema.rs:230-262) excludes the integrity field, so re-verifying a loaded, chain-linked row validates correctly. To make the chain itself an integrity boundary rather than mere tamper-evidence, additionally key it (HMAC with a store secret) or anchor the chain head with the sign-off authority's signature, since an unkeyed sha256 chain provides no protection once an editor recomputes it. Update main.rs:322-325 to pass the pubkey into open so disk rows face the same boundary as submitted rows, and correct the RowIntegrity/FOLLOWUPS docs to list the open-time attestation gap and full re-chaining as residuals.

[C5] HIGH  —  FileCorpus loads and serves persisted rows without ever checking the sign-off attestation, so an authority configured via with_authority never protects on-disk history
====================================================================================================
LOCATION: crates/corpus-client/src/store.rs:282-297 (open() sets authority: None and runs only verify_chain; with_authority() mutates the field post-load and re-validates nothing), :318-325 (query() never re-runs the gate), and the gate's only attestation entry point admit()/ensure_attested at :14-23 / :239,:330,:417 is reachable solely from submit() — never from open or query.

REASONING:
Traced the full failure path in current source. ensure_attested (gate.rs:104-119) is called ONLY from admit() (store.rs:14-23), which is called ONLY from the three submit() implementations (store.rs:239, 330, 417). It is never invoked on open()/load or on query(). FileCorpus::open() (store.rs:265-290) reads every row, runs verify_chain() (store.rs:282), and hard-codes authority: None (store.rs:286). with_authority() (store.rs:294-297) is a post-load builder that only sets self.authority; it re-validates nothing. query() (store.rs:318-325) calls fix_mappings directly with no gate. So `.open(path).with_authority(pk)` enforces attestation only on FUTURE submits; every row already on disk is served on the strength of the chain alone.

The chain is not a substitute defense. chain_hash (schema.rs:945-959) is a plain, unkeyed sha256 over (prev || canonical-row-without-integrity); verify_chain (store.rs:136-158) only checks that each row's stored prev/hash recompute. Anyone with write access to the JSONL file can therefore forge a fully chain-valid ResolvedConfirmed + SignOff::HumanConfirmed row with attestation: None and recompute the integrity links by hand, because no secret is involved. On the next authority-enabled run, open() succeeds (chain verifies), with_authority(pk) sets the key, the forged row is never run through ensure_attested, and query() serves it as a fix. This nullifies precisely the cryptographic half of the truth-admission boundary that the ed25519 attestation was built to provide ("the engine cannot mint a passing attestation"), for exactly the durable cross-run flywheel data that is most worth protecting. The with_authority doc at store.rs:292-294/198-200 explicitly promises "Require EVERY confirmed row to carry a valid sign-off attestation" — that invariant is false for all loaded rows.

I tried to refute it: (a) Does the chain stop the forgery? No — it is unkeyed and recomputable, so it stops only accidental/uncoordinated edits, not a deliberate write. (b) Is this merely intended cold-start migration? No — cold-start tolerance is a separate, legitimately-tested behavior (without_an_authority_an_unattested_confirmed_row_is_accepted), but here an authority IS configured at the time of query and the gate's own contract says such rows must be attested. (c) Is there a validating open path I missed? No — grep confirms open() is the only constructor and no open_with_authority exists. The reviewer rated this medium; I raise to high because it defeats the core integrity control for the persisted corpus and requires only file-write access (the exact threat the attestation defends against), with no chain barrier.

FIX:
Make the authority known before rows are validated and re-run ensure_attested over loaded rows. Add a validating constructor, e.g. `FileCorpus::open_with_authority(path, authority)` (or accept `Option<SignOffPublicKey>` in open()): after read+parse and verify_chain, if an authority is present, run `admit(row, &Some(authority))` (or at least `ensure_attested`) on every confirmed/resolved loaded row and return CorpusError::Storage/Gate (or quarantine the row out of fix_mappings) for any that lack a valid attestation. Critically, set self.authority BEFORE the loaded rows are considered usable, instead of the current `authority: None` + post-load with_authority mutation. Keep the cold-start path (no authority) unchanged. Optionally deprecate the post-open with_authority builder for FileCorpus so callers cannot reintroduce the gap.

[C6] CRITICAL  —  On-open admission gate is the keyless hash chain alone — attestation and evidence-integrity are never re-checked on rows loaded from disk, so a whole-file rewrite of forged confirmed fixes is served even with an authority configured
====================================================================================================
LOCATION: crates/corpus-client/src/store.rs:265-290 (FileCorpus::open — only verify_chain, never admit/ensure_attested); store.rs:136-158 (verify_chain); schema.rs:269-283 (keyless chain_hash); production wiring crates/support-agent/src/main.rs:322-337 (with_authority applied after open; query immediately follows)

REASONING:
Traced the full failure path in the real source and could not refute it.

1. FileCorpus::open (store.rs:267-282) parses every row and validates them ONLY via verify_chain(&rows) at store.rs:282. It never calls admit, ensure_evidence_integrity, or ensure_attested. A crate-wide grep confirms admit() is invoked at exactly three sites — store.rs:239/330/417, all the submit path — and nowhere on the load path.

2. verify_chain (store.rs:136-158) checks only the sha256 hash chain. chain_hash (schema.rs:269-283) is keyless: sha256(prev || canonical(row-without-integrity)), with prev seeded from "" (store.rs:146). It is therefore fully recomputable by anyone with write access to the corpus file. verify_chain returns Ok for any internally-consistent genesis-rooted chain, regardless of whether the rows carry an attestation.

3. The attestation field is Option with skip_serializing_if = Option::is_none (schema.rs:169-170), so a forged row simply omits it from JSON. An attacker fabricates ResolvedConfirmed/HumanConfirmed rows with verification set to a matching Pass (so the rows would pass ensure_evidence_integrity too, though that is never run) and attestation absent, then recomputes the chain. open() succeeds.

4. with_authority (store.rs:294-297) is applied in main.rs:323-325 AFTER FileCorpus::open(path)? at main.rs:322, and only gates future submit calls. It provides zero protection for the pre-existing on-disk rows.

5. main.rs:337 calls corpus.query immediately after open; query -> fix_mappings (store.rs:64-129) serves those forged resolved rows as retrievable CorpusPrimed fixes (main.rs:352-365), which are then preferred retrieval-first and can drive execution.

This defeats the entire MH-1 guarantee. The submit-path tests (authority_store_rejects_an_unattested_confirmed_row, store.rs:697) prove forgery is blocked on submit, but the rows that actually back today's retrieval enter via open and are admitted on the keyless chain alone. The RowIntegrity doc (schema.rs:98-104) calls out only tail truncation as the residual; whole-file re-genesis is the strictly more dangerous generalization the doc understates. Severity critical stands: it is a complete integrity/authenticity bypass of the signed-evidence boundary with a configured authority.

FIX:
See the fix field above.

[C7] HIGH  —  Reopened demotion is not run-deduplicated (asymmetric with confirmation counting) — a single reopen run replayed N times double-subtracts and silently buries a multiply-confirmed fix
====================================================================================================
LOCATION: crates/corpus-client/src/store.rs:107-108 (acc.reopened += 1) and the net at :121 (contributors.len().saturating_sub(acc.reopened)), asymmetric with confirmation dedup at :48-54/:104-105

REASONING:
Traced the concrete failure path in the real source.

Confirmation side (deduplicated): fix_mappings() at store.rs:104-105 inserts confirmation_key(...) into a HashSet. confirmation_key (store.rs:48-54) returns "run:{run_id}" for provenance rows, so two rows sharing a run_id collapse to one set entry. Test the_same_run_is_one_confirmation_not_many (store.rs:612-630) confirms: two submissions of run-A => confirmations == 1.

Reopen side (NOT deduplicated): the else branch at store.rs:107-108 runs `acc.reopened += 1` once per matching Reopened row, with no run_id awareness and never calling confirmation_key. The net at store.rs:121 is `confirmations = contributors.len() as u32 - acc.reopened` (saturating). So K duplicate reopen rows under one run_id subtract K, while K duplicate confirmations under one run_id add 0.

Exploit (matching the engine's own test a_reopened_outcome_demotes_the_fix at store.rs:850): run-A confirm + run-C confirm => contributors len 2; one run-B reopen => reopened 1 => net 1 (offered). But submitting the SAME run-B reopen TWICE => reopened 2 => net 0 => the mapping is dropped (confirmations>0 false at :122), despite two genuinely independent confirmations. A single reopen observation is thus amplified to cancel two independent confirmations; an appender needs only to duplicate one Reopened line N times to bury any plan out of retrieval.

Nothing refutes this:
- Gate (gate.rs:58-89) has zero run_id/dedup awareness; a Reopened label is non-resolved, so it skips the verdict/destructive checks and is admitted on a confirmed sign-off alone.
- With an authority configured, attestation_message (schema.rs:230-262) binds run_id, but two replays of the SAME reopen run produce byte-identical messages, so the same legitimate attestation verifies for both copies (ensure_attested, gate.rs:104-119). One legitimately-attested reopen run, replayed, suffices.
- verify_chain (store.rs:136-158) / submit (store.rs:327-354) just append; identical-content rows form valid sequential chain links (each hash folds in the differing prev), so duplicates pass integrity.

Second sub-claim also holds: because the reopen branch never calls confirmation_key, the circularity rule (store.rs:50, retrieval_first && primed_from contains plan_id) is never applied to reopens. A reopen whose own run is self-primed/circular still increments acc.reopened and nets against a genuine confirmation, even though by the engine's own independence logic a circular row carries no independent signal — so it should not be allowed to cancel one.

This is an integrity/availability defect on the core retrieval signal at the same append trust boundary the tamper-evidence/attestation work defends: it silently suppresses true, multiply-confirmed precedents. High is correct (it does not admit false truth, but it lets a single observation override the independence accounting that the rest of the change is built to enforce).

FIX:
Make demotion symmetric with confirmation counting — dedup reopens by the same independence key and subtract the count of DISTINCT reopen runs.

Replace the u32 `reopened` counter with a HashSet, and key reopens the same way confirmations are keyed:

  struct Acc {
      signature: FaultSignature,
      plan: common::Plan,
      contributors: HashSet<String>,
      reopens: HashSet<String>,   // was: reopened: u32
  }

In the loop (store.rs:103-109):

  if resolved {
      if let Some(key) = confirmation_key(row, &plan_id, index) {
          acc.contributors.insert(key);
      }
  } else {
      // dedup reopens by the same independence key as confirmations
      acc.reopens.insert(match &row.provenance {
          Some(p) => format!("run:{}", p.run_id),
          None => format!("row:{index}"),
      });
  }
